# 面试核心评测指标

这份报告只保留面试最值得讲的指标。完整离线评测结果见
`evals/reports/latest.md` 和 `evals/reports/latest.json`。

## 核心结论

当前离线评测覆盖：

- RAG 知识库问答：15 条
- Agent 工具调用：12 条
- AIOps 故障诊断：9 条

综合分为 `0.8781`。这个分数不是为了证明系统完美，而是说明项目已经有一套可复现的质量基线，并且能暴露具体优化方向。

## 最重要的 6 个指标

| 能力 | 指标 | 当前结果 | 面试解释 |
|---|---:|---:|---|
| RAG 召回覆盖 | Hit@3 | 100.00% | 15 个问题的正确文档都能进入 Top3，说明知识库召回没有漏掉关键文档。 |
| RAG 排序质量 | Hit@1 | 86.67% | 13/15 个问题首位命中，CPU 类问题有排序偏差，说明还需要 rerank 或领域权重。 |

| AIOps 根因判断 | Root Cause Hit@1 | 88.89% | 9 个故障案例中 8 个第一根因命中，网络/DNS 类故障仍是短板。 |
| AIOps 报告可执行性 | Action Coverage | 88.89% | 大多数诊断能给出预期处理动作，但网络/DNS 案例缺少 DNS/LB/流量切换建议。 |

## 当前暴露的问题

1. **RAG 排序仍需优化**
   - `rag_cpu_001` 和 `rag_cpu_003` 的正确文档在 Top3 内，但不是 Top1。
   - 后续可以加入告警名权重、标题权重、embedding rerank 或 cross-encoder rerank。

2. **工具选择还不够精准**
   - 当前 Exact Match 只有 `50.00%`。
   - 问题集中在：该查知识库时漏查、只查知识库的问题误查日志、慢响应问题漏查系统资源。
   - 后续可以优化工具描述、加工具路由器，或者基于真实 trace 训练/评估 tool policy。

3. **AIOps 场景覆盖不够完整**
   - 网络/DNS/负载均衡类服务不可用被误判成数据库连接失败。
   - 后续需要补网络诊断工具和对应排查知识：DNS 解析、LB 健康检查、跨地域超时、流量切换。

## 面试表达建议

可以这样讲：

> 我没有只看 Agent 能不能回答，而是拆成三层评测：RAG 是否召回正确文档、Agent 是否选对工具、AIOps 是否命中根因并给出可执行建议。当前 RAG Hit@3 是 100%，说明召回覆盖没问题；Hit@1 是 86.67%，暴露出排序仍需优化。工具调用 Micro F1 是 85.71%，说明工具选择总体可用，但 Exact Match 只有 50%，还需要优化工具路由。AIOps Root Cause Hit@1 是 88.89%，失败点集中在网络/DNS 类故障，这给后续扩展诊断工具提供了明确方向。

## 不建议过度强调的指标

- `Overall score = 0.8781` 可以提，但不建议放在第一位。
- `Required Term Recall = 1.0` 和 `Answer Term Coverage = 1.0` 只是说明 Top3 上下文材料足够，不等价于真实 LLM 回答完全正确。
- `Evidence Coverage = 1.0` 是因为离线案例的证据字段比较明确，真实线上还要接入日志和监控 trace 验证。
